## Why Care?

The Lossless inline-citation contract assumes one marker per cite point.
Three identical hex markers stacked together — `[^tv2rct] [^tv2rct] [^tv2rct]` —
is not how a human writes a citation; it is the visible residue of an
upstream LLM that emitted `[1, 1, 1]` or `[1][1][1]` because it wanted to
weight the same source heavily for one claim. The hex-conversion path
faithfully expanded each numeric occurrence to a hex marker and dutifully
preserved the repetition. The result reads as broken prose.

This pathology is distinct from the one `Dedupe Citations by URL` solves.
URL-dedupe consolidates *different* hex IDs whose reference definitions
point at the same URL — a reference-section problem. Inline-repetition is
the *same* hex ID repeated in body prose with horizontal whitespace
between occurrences — a different problem requiring a different transform.

## What Was Built / Updated

### New helper — `citationService.collapseInlineRepeats`

Single regex: `/(\[\^([a-z0-9]+)\])(?:[ \t]+\[\^\2\])+/g`. The
backreference `\2` ensures only same-hex runs collapse. Horizontal-
whitespace-only separator (`[ \t]+`, not `\s+`) keeps the pass line-scoped
so cross-paragraph repetition — which may legitimately cite the same
source from each bullet or paragraph — is preserved.

```ts
public collapseInlineRepeats(content: string): { content: string; collapsed: number }
```

Returns the rewritten content plus a count of collapsed runs (one count
increment per collapsed run, regardless of how many identical markers
were in the run).

### Wired into Convert-All-to-Hex

`main.ts` `convertAllCitations` now runs the dedupe pass after the
hex-conversion loop. The pass runs even when no conversions occurred —
a document may already be hex-only but still carry repetition runs from
a prior LLM paste, and Convert-All is the natural surface where the user
expects one-stop cleanup.

The post-conversion notice now reflects both transforms when both fire:

```
converted 12 citations to hex; collapsed 3 inline repetitions
```

### Wired into the LLM-paste pipeline

`LlmCitationParserService.transform` runs the same regex on the joined
output before returning. `TransformStats` gained an
`inlineRepetitionsCollapsed` field, and the consuming modals
(`PasteLlmContentModal`, `LlmCitationsModal`) surface the count in their
post-action notices when non-zero.

This closes the loop at the source: a paste that produces
`[1, 1, 1]` → `[^a] [^a] [^a]` now collapses to `[^a]` in the same
transform, before the editor ever sees the duplicate markers.

### New standalone command

`Dedupe Inline Citation Repetitions` (id:
`dedupe-inline-citation-repetitions`) runs the helper on the active
file. It exists because the auto-pass inside Convert-All-to-Hex only
helps documents that pass through Convert-All going forward — files
already in the vault with stale repetition runs need a one-shot cleanup
command. This is also the command users will reach for when they spot
the pathology mid-document and don't want to re-run the whole conversion.

## What Changed in Approach

| Pattern rejected | Pattern adopted |
|---|---|
| Hide the helper inside one modal flow | Single helper, three call sites (convert-all, paste pipeline, standalone command) |
| Use `\s+` as the separator | Use `[ \t]+` so newlines are not crossed |
| Generalize to "any duplicate citations" | Same-hex runs only; URL-dedupe stays focused on its own concern |
| Skip stats plumbing | Track `inlineRepetitionsCollapsed` in `TransformStats` so users see what changed |

The generalizable point: **citation-cleanup transforms should be cheap
to compose and visible in their effects.** Three call sites, one
regex, one stat, one notice phrase. Anything more would be premature.

## Verification

`pnpm build` passed: TypeScript clean, ESLint clean, esbuild production
build clean. Sanity-checked the regex against the example string the
user reported — `er. [^tv2rct] [^tv2rct] [^tv2rct]` collapses to
`er. [^tv2rct]`, with non-citation prose ("er.") preserved.

## Open Items

- **No tests committed.** The regex is simple enough to verify by
  inspection, but a future addition to `scripts/parse-llm-citations.mjs`
  exercising the dedupe would harden against future regex tweaks.
- **Punctuation-separator dedupe not added.** `[^hex], [^hex]` is left
  unchanged. If users encounter this in real content, it can be added
  to the regex as `(?:[ \t,;]+\[\^\2\])+` — but the conservative
  whitespace-only form is the right starting point.
- **Cross-line dedupe not added.** Same-hex on consecutive lines stays
  put. This is intentional: each bullet or list item legitimately may
  cite the same source.
- **Reference-definition repetition not addressed.** `[^abc]: ...`
  duplicate definitions are a separate pathology that should surface as
  a flag, not be silently collapsed.
